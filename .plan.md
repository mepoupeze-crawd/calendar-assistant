# Calendar Assistant — Feature Plan: Contacts / Guest Resolution Fix

## Type: BUG FIX (existing partially-built feature)

## What We're Fixing

The guest resolution flow (parse participant name → Google People lookup → pick contact → add as guest) has two bugs that make it unusable:

1. **Button layout broken for multiple contacts** — all pick buttons appear in a single horizontal row, Telegram squeezes them and they become unreadable/untappable on mobile.
2. **Indistinguishable failure messages** — when contacts lookup crashes (any error), user gets the same "type the email" prompt as when the contact genuinely doesn't exist — impossible to tell if the integration is working.

---

## Desired Behavior

When user describes an event mentioning a person (e.g., "Reunião com Aline amanhã às 14h"):

1. Parser extracts participant: `{ name: "Aline", email: null }`
2. Bot looks up "Aline" in Google People API
3. **1 match** → auto-resolve silently, continue to preview
4. **Multiple matches** → show each option as its **own button row** (not all in one row) + "✍️ Digitar email" as last row
5. **0 matches** → ask user "Qual o email de Aline?"
6. **Lookup error** → tell user "couldn't access contacts" (different from "not found")

---

## Root Cause Analysis

### Bug 1: All contact buttons in one row (main issue)

`sendMessage` in `bot.ts:184` always renders all buttons in a single array row:
```ts
inline_keyboard: [buttons.map(btn => ({ text: btn.text, callback_data: btn.callback_data }))]
//              ^ wraps everything in ONE row
```

When `resolveParticipantsAndPreview()` in `telegram-calendar.ts` builds the contact pick response:
```ts
buttons: [
  { text: 'Aline Silva',  callback_data: 'pick_contact_0' },
  { text: 'Aline Costa',  callback_data: 'pick_contact_1' },
  { text: '✍️ Digitar email', callback_data: 'pick_contact_manual' },
]
```

This becomes `inline_keyboard: [[Aline Silva, Aline Costa, ✍️ Digitar email]]` — three buttons crammed into one horizontal row. On mobile this is nearly impossible to use.

**Required result:**
```
inline_keyboard: [
  [{ text: 'Aline Silva',      callback_data: 'pick_contact_0' }],
  [{ text: 'Aline Costa',      callback_data: 'pick_contact_1' }],
  [{ text: '✍️ Digitar email', callback_data: 'pick_contact_manual' }],
]
```

Note: The Confirm/Cancel/Edit preview buttons must stay in ONE row — the fix must not break those.

### Bug 2: Lookup error is indistinguishable from "not found"

`lookupContactsByName` in `contacts.ts`:
```ts
} catch (err) {
  console.warn('[Contacts] Lookup failed...');
  return [];  // same as "0 results" — caller can't tell the difference
}
```

`resolveParticipantsAndPreview` treats `[]` as "no contact found" regardless of cause, so any error silently produces "Qual o email?" — user has no idea if their contacts integration works.

**Fix:** Return a distinct `{ contacts: Contact[]; error: boolean }` from `lookupContactsByName`, so the caller can show a different message when there's an error vs. a genuine no-match.

## Changes Required

### Modified Components

**`src/lib/calendar/contacts.ts`**
- Change return type: `Promise<Contact[]>` → `Promise<{ contacts: Contact[]; error: boolean }>`
- On success: `{ contacts: results, error: false }`
- On catch: `{ contacts: [], error: true }` (and log the error as before)

**`src/handlers/telegram-calendar.ts`**
- Update `TelegramResponse.buttons` type: `Array<{text, callback_data}>` → `Array<Array<{text, callback_data}>>` (rows)
- Update `resolveParticipantsAndPreview()`:
  - Update call to `lookupContactsByName` → destructure `{ contacts, error }`
  - On `error: true` → different message: "⚠️ Não consegui acessar seus contatos.\n\nQual o email de <b>{name}</b>?"
  - Contact options buttons: each option as its own row: `[[opt0], [opt1], ..., [✍️ Digitar email]]`
  - Update `lookupContactsByName` import to handle new return type
- Update `previewEvent()`:
  - Preview buttons stay in ONE row: `[[Confirmar, Cancelar, Editar]]`

**`src/bot.ts`**
- Update `sendMessage()` parameter: `buttons?: Array<{...}>` → `buttons?: Array<Array<{...}>>`
- Update `sendMessage()` body: `inline_keyboard: buttons` (pass rows directly — remove the wrapping `[...map]`)
- Find all ad-hoc `sendMessage(chatId, text, [{ text, callback_data }])` calls and wrap each button in its own array: `[[{ text, callback_data }]]`
- No changes to state machine logic — only button format

### Unchanged Components
- `src/lib/calendar/creator.ts` — correct, not touched
- `src/lib/calendar/parser.ts` — not touched
- `src/lib/calendar/validator.ts` — not touched
- `src/lib/calendar/previewer.ts` — not touched
- `scripts/authorize.mjs` — not touched

## Type Contract Change

```ts
// contacts.ts — BEFORE
export async function lookupContactsByName(name: string): Promise<Contact[]>

// contacts.ts — AFTER
export async function lookupContactsByName(name: string): Promise<{ contacts: Contact[]; error: boolean }>
```

```ts
// TelegramResponse — BEFORE
buttons?: Array<{ text: string; callback_data: string }>

// TelegramResponse — AFTER
buttons?: Array<Array<{ text: string; callback_data: string }>>
```

```ts
// sendMessage — BEFORE
inline_keyboard: [buttons.map(btn => ({ text: btn.text, callback_data: btn.callback_data }))]

// sendMessage — AFTER
inline_keyboard: buttons  // already rows — pass directly
```
