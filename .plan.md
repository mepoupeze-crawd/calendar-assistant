# Calendar Assistant — Feature Plan: Preview Visual Redesign

## Type: NEW FEATURE (incremental)

## What We're Adding

Redesign da mensagem de preview de eventos no Telegram: de uma lista de campos com asteriscos literais para um card visual limpo usando HTML (alinhado com o `parse_mode: 'HTML'` já usado pelo bot).

**Bug implícito corrigido:** o previewer gera MarkdownV2 (`**bold**`, `\_italic\_`) mas o bot envia com `parse_mode: 'HTML'` — os asteriscos aparecem literalmente para o usuário em vez de renderizar como negrito.

**Novo visual (HTML):**
```
📅 <b>Manu - 1 aninho</b>

🗓 Sábado, 07/03/2026  ·  ⏰ 16:00
📍 Rua Desembargador do Vale, 330 – Salão de Festas
👥 João, Maria

⚠️ Conflito: Reunião de equipe (15:00–17:00)    ← só se houver

Confirme o evento:
```

---

## Why This Feature

- O usuário vê `**Título:**` com asteriscos literais — a formatação não renderiza
- A lista de campos rotulados (`**Título:**`, `**Data:**`) é verbosa e difícil de escanear
- Um card com ícones e hierarquia visual é mais rápido de ler numa interface de chat
- Zero dependências novas, zero mudanças de arquitetura

## Existing Context

**Onboarding report**: `E:/Documentos/IA/_teste_agents/calendar-assistant-vAgents/.onboard.md`
**Current architecture**: Single-process polling bot; `parse_mode: 'HTML'` em `bot.ts:sendMessage()`
**Current stack**: TypeScript 5, Node.js 22, ts-node

## Root Cause

`previewer.ts` gera texto com syntax MarkdownV2:
- `**Título:**` → deveria ser `<b>Título</b>` em HTML
- `_Confirme, edite ou cancele:_` → deveria ser `<i>Confirme o evento:</i>`
- `escapeMarkdown()` escapa para MarkdownV2 (ex: `\-`) mas o bot usa HTML

---

## Changes Required

### Modified Components

**`src/lib/calendar/previewer.ts`** — único arquivo que muda:

1. **`buildPreviewText()`** — reescrever para gerar HTML em vez de MarkdownV2:
   - Título como `<b>título</b>` em destaque no topo
   - Data + horário na mesma linha com ícones: `🗓 Sábado, 07/03/2026  ·  ⏰ 16:00`
   - Local em linha própria com `📍`
   - Participantes em linha própria com `👥` (omitir se vazio)
   - Descrição em linha própria com `📝` (omitir se vazia)
   - Conflitos com `⚠️` e nome do evento em conflito
   - Footer: `Confirme o evento:` (sem italics desnecessário)

2. **`escapeMarkdown()` → `escapeHtml()`** — substituir por escape HTML:
   - Só precisa escapar: `&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;`
   - Remover todos os escapes de MarkdownV2

3. **Manter inalterado:**
   - `generateEventId()` — não muda
   - `buildInlineKeyboard()` — não muda
   - `ConflictInfo` interface — não muda
   - `buildEditMenu()` — não muda (pode receber HTML também se necessário)
   - Assinatura de `generatePreview()` — não muda

### Unchanged Components

- `src/bot.ts` — `parse_mode: 'HTML'` já está correto, não muda
- `src/handlers/telegram-calendar.ts` — não muda
- `src/lib/calendar/types.ts` — não muda
- Todos os outros módulos de `src/lib/calendar/`

## New Format Spec (HTML)

```
📅 <b>{título}</b>

🗓 {dia-da-semana}, {DD/MM/YYYY}  ·  ⏰ {HH:MM}[–{HH:MM}]
                                          ← "O dia todo" se all_day
📍 {local}                               ← omitir linha se ausente
👥 {participante1}, {participante2}       ← omitir linha se ausente
📝 {descrição}                           ← omitir linha se ausente, truncar em 200 chars

⚠️ Conflito: <b>{título-conflito}</b> ({start}–{end})   ← omitir bloco se sem conflitos

Confirme o evento:
```

**Regras HTML a seguir:**
- Só tags permitidas pelo Telegram HTML mode: `<b>`, `<i>`, `<code>`, `<a>`
- Escapar conteúdo do usuário: `&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;`
- Nunca usar `&quot;` ou `&#` — Telegram não aceita todas as entidades HTML

## Integration Points

- `telegram-calendar.ts` chama `generatePreview(event, conflicts)` → consome `preview.text` → passa para `sendMessage(chatId, text, buttons)`
- `sendMessage` em `bot.ts` usa `parse_mode: 'HTML'` — compatível com o novo formato
- Nenhum contrato de interface muda

---

## Acceptance Criteria

1. Mensagem de preview renderiza negrito, ícones e layout visual no Telegram (sem asteriscos literais)
2. Todos os campos opcionais (local, participantes, descrição) são omitidos quando ausentes
3. Conflitos são exibidos quando presentes, omitidos quando ausentes
4. Caracteres especiais no título (`-`, `&`, `<`) não quebram a mensagem HTML
5. Eventos all-day mostram "O dia todo" em vez de horário
6. 40/40 testes Jest continuam passando após a mudança

## Risks and Mitigations

| Risco | Impacto | Probabilidade | Mitigação |
|-------|---------|---------------|-----------|
| Tag HTML inválida enviada ao Telegram → mensagem falha | Alto | Baixo | `escapeHtml()` garante que conteúdo do usuário nunca injeta tags; testar com títulos com `<` e `&` |
| Regressão nos testes de pipeline | Médio | Baixo | `npm test` antes de commitar; testes de previewer verificam `text` gerado |
| Telegram rejeitar entidades HTML não suportadas | Médio | Baixo | Usar apenas `<b>`, `<i>` — tags documentadas na Telegram Bot API |
