# Calendar Assistant — Deployment Plan: GCP e2-micro + GitHub Actions CI/CD

## Type: DEPLOYMENT (feature for existing project)

## What We're Deploying

O bot Telegram do Calendar Assistant rodando 24/7 em uma VM gratuita no GCP, gerenciado pelo systemd (reinicia automaticamente se cair). CI/CD via GitHub Actions: push para `main` → deploy automático sem intervenção manual.

**Custo estimado: $0/mês** (tudo dentro do free tier do GCP)

**Novo em produção:**
- VM e2-micro no GCP rodando o bot via systemd (auto-restart)
- Deploy automático a cada push para `main` via GitHub Actions
- Logs via `journalctl -u calendar-bot -f`
- Zero mudança no `src/bot.ts` ou na lógica do bot

## Architecture Decision: Compute Engine e2-micro

| Opção | Custo | Gestão | Adequação para polling bot |
|---|---|---|---|
| **e2-micro + systemd** ✅ | **$0/mês (free tier)** | SSH para emergências | Perfeita — processo contínuo, sem restrição de porta |
| Cloud Run (min-instances=1) | ~$5-8/mês | Zero | Requer servidor HTTP no processo |
| Cloud Run Jobs | $0 | Zero | Não serve — não roda contínuo |

**Escolha: Compute Engine e2-micro**
- Free tier: 1 e2-micro/mês em us-central1/us-east1/us-west1 para sempre
- Systemd: reinicia o bot automaticamente se travar ou cair
- Nenhuma mudança no código do bot
- CI/CD via GitHub Actions: push → deploy em ~30 segundos

### FASE 1 — Criar e configurar a VM (one-time, ~15 min)

**1.1 — Criar a VM e2-micro no free tier**
```bash
gcloud compute instances create calendar-assistant \
  --machine-type=e2-micro \
  --zone=us-central1-a \
  --image-family=debian-12 \
  --image-project=debian-cloud \
  --boot-disk-size=20GB \
  --boot-disk-type=pd-standard \
  --tags=calendar-bot
```

**1.2 — Reservar IP estático** (para o CI/CD sempre saber o endereço)
```bash
gcloud compute addresses create calendar-bot-ip \
  --region=us-central1

# Associar à VM
gcloud compute instances delete-access-config calendar-assistant \
  --access-config-name="External NAT" \
  --zone=us-central1-a

gcloud compute instances add-access-config calendar-assistant \
  --access-config-name="External NAT" \
  --address=$(gcloud compute addresses describe calendar-bot-ip --region=us-central1 --format='value(address)') \
  --zone=us-central1-a

# Anotar o IP — vai ser usado nas Fases 2 e 3
gcloud compute addresses describe calendar-bot-ip --region=us-central1 --format='value(address)'
```

**1.3 — SSH na VM**
```bash
gcloud compute ssh calendar-assistant --zone=us-central1-a
```

**1.4 — Instalar Node 22 e Git** (dentro da VM)
```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt-get install -y nodejs git
node --version   # deve mostrar v22.x
```

**1.5 — Criar usuário dedicado e clonar o repo**
```bash
sudo useradd -m -s /bin/bash calendar
sudo mkdir -p /opt/calendar-assistant
sudo chown calendar:calendar /opt/calendar-assistant
```

**Configurar deploy key** (para a VM fazer `git pull` do repo privado):
```bash
# Ainda dentro da VM, virar o usuário calendar
sudo su - calendar

# Gerar chave SSH para deploy
ssh-keygen -t ed25519 -C "calendar-assistant-vm" -f ~/.ssh/deploy_key -N ""

# Mostrar a chave pública — vai adicionar no GitHub na Fase 2
cat ~/.ssh/deploy_key.pub

# Configurar o SSH para usar essa chave no GitHub
cat >> ~/.ssh/config << 'EOF'
Host github.com
  IdentityFile ~/.ssh/deploy_key
  StrictHostKeyChecking no
EOF

# Clonar o repo (substituir pela URL SSH do seu repo)
git clone git@github.com:SEU_USUARIO/calendar-assistant.git /opt/calendar-assistant

cd /opt/calendar-assistant
npm install

# Sair do usuário calendar
exit
```

**1.6 — Criar o arquivo `.env` na VM**
```bash
sudo -u calendar nano /opt/calendar-assistant/.env
```

Conteúdo do `.env` (preencher com valores reais):
```
TELEGRAM_BOT_TOKEN=
OPENAI_API_KEY=
ALLOWED_CHAT_ID=
GOOGLE_OAUTH_CLIENT_ID=
GOOGLE_OAUTH_CLIENT_SECRET=
GOOGLE_OAUTH_REFRESH_TOKEN=
GOG_ACCOUNT=
CALENDAR_OWNER_EMAIL=
```

Proteger o arquivo:
```bash
sudo chmod 600 /opt/calendar-assistant/.env
sudo chown calendar:calendar /opt/calendar-assistant/.env
```

**1.7 — Instalar e ativar o serviço systemd**
```bash
# O arquivo deploy/calendar-bot.service está no repo (criado na Fase 2 do código)
sudo cp /opt/calendar-assistant/deploy/calendar-bot.service /etc/systemd/system/

sudo systemctl daemon-reload
sudo systemctl enable calendar-bot
sudo systemctl start calendar-bot

# Verificar
sudo systemctl status calendar-bot
```

**1.8 — Configurar sudo sem senha para o usuário calendar** (necessário para o CI/CD reiniciar o serviço)
```bash
echo 'calendar ALL=(ALL) NOPASSWD: /bin/systemctl restart calendar-bot, /bin/systemctl status calendar-bot' \
  | sudo tee /etc/sudoers.d/calendar-bot
sudo chmod 440 /etc/sudoers.d/calendar-bot
```

**Sair da VM:**
```bash
exit
```

**@devops** implementa:

**2.2 — `deploy/calendar-bot.service`** (unit file do systemd)
```ini
[Unit]
Description=Calendar Assistant Telegram Bot
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=calendar
WorkingDirectory=/opt/calendar-assistant
EnvironmentFile=/opt/calendar-assistant/.env
ExecStart=npx ts-node -P tsconfig.test.json src/bot.ts
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal
SyslogIdentifier=calendar-bot

[Install]
WantedBy=multi-user.target
```

**2.3 — `.github/workflows/deploy.yml`** (CI/CD)
```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to VM
        uses: appleboy/ssh-action@v1.2.0
        with:
          host: ${{ secrets.VM_HOST }}
          username: calendar
          key: ${{ secrets.VM_SSH_KEY }}
          script: |
            cd /opt/calendar-assistant
            git pull origin main
            npm install --omit=dev
            sudo systemctl restart calendar-bot
            sleep 5
            sudo systemctl status calendar-bot --no-pager -l
```

**Commitar e fazer push:**
```bash
git add deploy/calendar-bot.service .github/workflows/deploy.yml package.json package-lock.json
git commit -m "feat: add systemd service and GitHub Actions CI/CD"
git push origin main
```

### FASE 3 — Configurar CI/CD no GitHub (one-time)

**3.1 — Adicionar deploy key ao repositório GitHub**

No GitHub → repo → Settings → Deploy keys → Add deploy key:
- Title: `calendar-assistant-vm`
- Key: (colar o conteúdo de `~/.ssh/deploy_key.pub` da VM — copiado no passo 1.5)
- Allow write access: **NÃO** (read-only é suficiente para `git pull`)

**3.2 — Gerar chave SSH para o GitHub Actions SSH na VM**

Local (no seu computador):
```bash
ssh-keygen -t ed25519 -C "github-actions-deploy" -f ./gh-actions-key -N ""
# Isso cria: gh-actions-key (privada) e gh-actions-key.pub (pública)
```

Adicionar a chave pública ao `authorized_keys` da VM:
```bash
# Copiar o conteúdo de gh-actions-key.pub
cat gh-actions-key.pub

# SSH na VM e adicionar
gcloud compute ssh calendar-assistant --zone=us-central1-a
sudo -u calendar bash -c 'cat >> ~/.ssh/authorized_keys' << 'EOF'
CONTEUDO_DA_CHAVE_PUBLICA
EOF
exit
```

**3.3 — Adicionar GitHub Secrets ao repositório**

No GitHub → repo → Settings → Secrets and variables → Actions → New repository secret:

| Nome | Valor |
|---|---|
| `VM_HOST` | IP estático da VM (copiado no passo 1.2) |
| `VM_SSH_KEY` | Conteúdo do arquivo `gh-actions-key` (chave PRIVADA) |

Apagar os arquivos de chave locais após salvar:
```bash
rm gh-actions-key gh-actions-key.pub
```

## Risks and Mitigations

| Risco | Impacto | Probabilidade | Mitigação |
|-------|---------|---------------|-----------|
| `.env` exposto (sem gitignore) | Alto | Baixo | `.gitignore` já exclui `.env`; `.env` só existe na VM, nunca no repo |
| Chave privada SSH (`gh-actions-key`) vazada | Alto | Baixo | Apagar localmente após salvar no GitHub Secrets; @security valida |
| IP da VM muda (ephemeral) | Médio | Baixo | IP estático reservado no passo 1.2; free quando a VM está rodando |
| Refresh token revogado | Alto | Baixo | Token OAuth não expira por tempo. Rodar `authorize.mjs` localmente e atualizar `.env` na VM |
| npm install falha no CI/CD | Médio | Baixo | `npm install --omit=dev` apenas instala deps de produção; sem Next.js depois de 2.1 |
| VM cai (raro no GCP) | Médio | Muito Baixo | systemd `Restart=always`; VM e2-micro tem SLA 99.5% |
| Regressão no bot via deploy automático | Médio | Baixo | Rodar `npm test` localmente antes de push; rollback = `git revert + push` |
