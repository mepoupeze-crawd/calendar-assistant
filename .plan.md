# Calendar Assistant — Feature Plan: Multi-User Authorization

## Type: NEW FEATURE (incremental)

## What We're Adding

Allow a second Telegram user (the owner's wife) to send event descriptions to the same bot
and have them created on the shared Google Calendar. Currently the bot only accepts messages
from a single `ALLOWED_CHAT_ID`. This feature extends auth to a comma-separated list of allowed
chat IDs (`ALLOWED_CHAT_IDS`), while keeping full backward compatibility.

New capabilities:
- Wife messages the bot in her own private Telegram chat to create events on the shared Google Calendar
- Each user has their own independent conversation session (no state bleed between users)
- Unknown users are still silently rejected

---

## Why This Feature

Owner and wife share a Google Calendar. Currently only the owner can create events via the bot.
This feature removes that limitation with minimal code change: a 10-line auth extension.

## Changes Required

### Modified Components

**`src/bot.ts` (lines 20-27)**

Change single-ID auth to a Set-based allow-list:

```ts
// BEFORE
const ALLOWED_CHAT_ID = process.env.ALLOWED_CHAT_ID || '7131103597';
console.log(`[Bot] Configured to respond only in chat: ${ALLOWED_CHAT_ID}`);
// check: if (chatId !== ALLOWED_CHAT_ID) { ... }

// AFTER
const _rawAllowed = process.env.ALLOWED_CHAT_IDS || process.env.ALLOWED_CHAT_ID || '7131103597';
const ALLOWED_CHAT_IDS = new Set(_rawAllowed.split(',').map(id => id.trim()).filter(Boolean));
console.log(`[Bot] Configured to respond only in chats: ${[...ALLOWED_CHAT_IDS].join(', ')}`);
// check: if (!ALLOWED_CHAT_IDS.has(chatId)) { ... }
```

Backward compatibility: if only `ALLOWED_CHAT_ID` is set in `.env`, it still works.

**`.env.example`**

Add multi-user env var documentation:

```env
# Single user (legacy - still supported):
# ALLOWED_CHAT_ID=your_chat_id

# Multiple users (new - preferred):
ALLOWED_CHAT_IDS=owner_chat_id,wife_chat_id
```

## How to Find the Wife's chat_id

Before deploying, the wife's Telegram `chat_id` is needed.

**Option A - userinfobot (easiest):**
1. Wife opens Telegram and messages @userinfobot
2. It replies with her user ID - that is her `chat_id`

**Option B - from bot logs (temporary):**
1. Temporarily add `console.log('[Auth] Message from:', chatId);` before the auth check in `bot.ts`
2. Wife sends any message to the bot
3. Owner reads the bot logs, copies the ID
4. Remove the temporary log line

Then update `.env`:
```
ALLOWED_CHAT_IDS=7131103597,<wife_chat_id>
```
Restart the bot.

## Acceptance Criteria

1. Wife sends an event description to the bot and the event is created on the shared Google Calendar
2. Wife's session is independent from owner's (different clarification flow, no state bleed)
3. Unknown chat IDs are still silently rejected (no response sent)
4. Owner's existing flow is completely unchanged
5. Bot startup logs show all allowed chat IDs (not just one)
6. If only `ALLOWED_CHAT_ID` is set (no `ALLOWED_CHAT_IDS`), bot still works correctly
7. 40/40 tests continue passing
